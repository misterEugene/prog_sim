# Архитектура и решения по реализации

Документ описывает целевую архитектуру платформы, ключевые решения и обоснования.
Дополняй его, когда принимаешь новое нетривиальное решение.

## Концепция

Локальная (без сервера/сборки/библиотек) платформа для обучения детей коду.
**Один** предустановленный урок, зашитый в `js/lesson-data.js` (объект `lesson`).
Создание новых уроков через UI **не предусмотрено** — это намеренное упрощение.

## Структура исходников (модули js/ и css/)

Раньше вся логика жила в одном `main.js` (~2380 строк), а стили — в одном
`style.css` (~840 строк). Файлы разнесены по модулям для читаемости. **Поведение
не менялось** — код перенесён байт-в-байт (нарезка по диапазонам строк, не
перепечатывание; см. PITFALLS про trailing whitespace).

**Почему классические скрипты, а НЕ ES-модули.** Платформа должна открываться
прямо в браузере, в т.ч. из `file://`. ES-модули (`type="module"`/`import`)
требуют HTTP (CORS блокирует `file://`), поэтому модули — обычные `<script defer>`.
Связь между ними — через **общую глобальную лексическую область**: top-level
`const`/`let`/`function` одного классического скрипта видны функциям другого
(в пределах одного realm). Это работает, потому что все межмодульные обращения
происходят ВНУТРИ функций, вызываемых на `DOMContentLoaded`, — к этому моменту
все `defer`-скрипты уже выполнены. На верхнем уровне модулей нет обращений к
чужим биндингам (только объявления + регистрация `init` в `js/main.js`), поэтому
порядок подключения для корректности некритичен; он задан логически, `js/main.js`
— последним.

Порядок `<script>` (в `index.html`): `emmet` → `lesson-data` → `dom` → `markdown`
→ `storage` → `editor` → `highlight` → `swatches` → `history` →
`emmet-integration` → `lesson-render` → `console` → `preview` → `zip` → `toast`
→ `reset` → `tabs` → `layout` → `dev-cheat` → `main`. (`dev-cheat.js`
подключён ТОЛЬКО в `index.html`, в `index2.html` его нет.)

### Раскладка из трёх колонок с перетаскиваемыми разделителями (VS Code-стиль)

Интерфейс перестроен: сверху **`.topbar`** (заголовок урока + общие кнопки:
Запустить / Открыть / Начать заново / Подсказка / Скачать + «Свернуть задание»),
ниже **`.layout`** — три колонки, разделённые перетаскиваемыми `.splitter`:

```
┌──────────────────────── .topbar (кнопки) ───────────────────────────┐
├───────────────┬─┬───────────────────┬─┬────────────────────────────┤
│ #col-task     │║│ #col-editors      │║│ #col-preview               │
│ 📖 Урок       │║│ вкладки + редактор │║│ Результат (iframe) +       │
│ (markdown)    │║│ (index/css/js)    │║│ консоль                    │
└───────────────┴─┴───────────────────┴─┴────────────────────────────┘
                 ↑ #split-1            ↑ #split-2  (col-resize)
```

**Ширина — через `flex-grow`.** Колонки `.col { flex-grow:1; flex-basis:0 }`,
поэтому при `flex-basis:0` ширина пропорциональна `flex-grow` (по умолчанию все
три равны). `makeResizable(splitter,left,right)` на перетаскивании меняет
`flex-grow` ТОЛЬКО двух соседних колонок (их сумма-`grow` сохраняется), как в
VS Code — остальные не двигаются. Минимальная ширина колонки — `COL_MIN_PX` (140px).

**Pointer Events + захват указателя.** Перетаскивание реализовано на
`pointerdown/move/up` с `setPointerCapture` — иначе при заходе курсора на `iframe`
превью события `pointermove` терялись бы. На время перетаскивания на `body`
вешается класс `.resizing` (`user-select:none` + `#preview{pointer-events:none}`).

**Сворачивание любой колонки.** В шапке каждой колонки — кнопка «свернуть»
(`.col-collapse`; у редактора она в ряду вкладок). `setCollapsed(col,true)`
добавляет класс `.collapsed` и ставит `flex:0 0 auto` — колонка превращается в
тонкую вертикальную полосу (`.col-reopen`, ширина 38px, подпись через
`writing-mode:vertical-rl`), клик по которой её разворачивает. Доля колонки
запоминается в `col._savedGrow` и восстанавливается при развороте; остальные
колонки сохраняют пропорции и заполняют освободившееся место. `updateSplitters()`
прячет разделитель, если соседняя колонка свёрнута (тянуть полосу незачем).
`toggleCol` не даёт свернуть **последнюю** развёрнутую колонку (иначе экран пуст).

**Сохранение раскладки.** `saveLayout()`/`loadLayout()` хранят доли колонок
(`growOf` — у свёрнутой берёт `_savedGrow`) и карту свёрнутости
`{task,editors,preview}` в отдельном ключе localStorage **`layoutPrefs`** (не
смешивается с `savedLessonCode`). Восстанавливается при загрузке (`initLayout`).

**Вкладки.** Задание больше НЕ вкладка (у него своя колонка) — `switchTab`
переключает только три файла кода (`index.html`/`style.css`/`main.js`). Механизм
вставки блоков не изменился: `insertAsUserInput` по-прежнему зовёт
`switchTab(file)`, чтобы сфокусировать нужный редактор перед `execCommand`.

## Функции main.js и связи между ними

- `markdownToHtml(md)` — мини-парсер Markdown (без библиотек).
- `renderDescription()` — кладёт `markdownToHtml(lesson.descriptionMd)` в `.markdown-body`.
- `loadFromLocalStorage()` → `{html,css,js}` | `null`.
- `saveToLocalStorage(html,css,js)` — пишет один ключ `savedLessonCode` (JSON).
- `buildDocument()` — собирает полный HTML-документ из текущих редакторов (стили +
  служебные скрипты). Общий для превью и вкладки просмотра.
- `updateIframe()` — `clearConsole()` + кладёт `buildDocument()` в `iframe.srcdoc`
  и зеркалит документ в localStorage (`savePreviewDoc`, ключ `previewDoc`).
- `savePreviewDoc(doc)` — пишет готовый документ в localStorage (try/catch).
- `openInNewTab()` — сохраняет свежий снимок и открывает `preview.html` в
  именованной вкладке (`window.open(..., 'shopPreview')`). Если popup заблокирован —
  тост с подсказкой.

### Отдельная вкладка просмотра — `preview.html` (живое зеркало через localStorage)

Кнопка «🔗 Открыть в новой вкладке» открывает **отдельный файл `preview.html`**, а
НЕ Blob-снимок. Почему: Blob-URL — статичен (не отражает правки) и при отзыве
(`revokeObjectURL`) вкладка «ломалась» после перезагрузки.

Механизм: на каждый «Запустить» платформа кладёт готовый документ в localStorage
(`previewDoc`). `preview.html` содержит полноэкранный `<iframe>`, читает `previewDoc`
в свой `srcdoc` и подписан на событие **`storage`** (приходит из другой вкладки того
же origin) → переотображается синхронно с «Запустить». Плюсы: отражает изменения
(по Run, как и встроенное превью), переживает перезагрузку вкладки, ничего не
«отзывается». Нет дублирования логики сборки — `preview.html` лишь показывает уже
собранный платформой документ. (Событие `storage` стабильно по HTTP; на `file://`
зависит от браузера, но первичная загрузка/перезагрузка работают.)
- `resetToTemplate()` — заполняет редакторы `initial*`, чистит localStorage, зовёт `updateIframe()`.
- `crc32(bytes)` / `buildZip(files)` — ручная сборка ZIP-архива (метод store, без библиотек).
- `downloadBlob(blob, filename)` — `<a download>` + клик + revokeObjectURL.
- `downloadProject()` — собирает три файла в ZIP и скачивает `project.zip`.
- `showHint()` — показывает тост `lesson.hint`, прячет через 5 с.
- `switchTab(tabId)` — переключает активную вкладку и видимый блок.

Поток данных:
```
input в textarea ──> saveToLocalStorage()         (iframe НЕ трогаем)
кнопка «Запустить» ──> updateIframe()             (читает текущие textarea)
загрузка страницы ──> loadFromLocalStorage() || initial* ──> renderDescription() ──> updateIframe()
```

## Ключевые решения

### Сборка документа для iframe
`updateIframe()` присваивает результат `iframe.srcdoc` (**`srcdoc`, а не
`document.write`** — см. PITFALLS). Две ветки:

- **Фрагмент** (в HTML-редакторе нет `</body>`): оборачиваем как раньше —
  `<!DOCTYPE html><html><head><style>CSS</style></head><body>HTML+scripts</body></html>`.
- **Полный документ** (ученик создал каркас через Emmet «!», есть `</body>`):
  используем его как есть, лишь внедряя `<style>CSS</style>` перед `</head>` и
  скрипты (CONSOLE_HOOK + JS) перед `</body>`. Иначе вышел бы документ, вложенный
  в `<body>` (двойные `<html>/<head>/<body>`).

Замены `</head>`/`</body>` делаются **функцией-replacer** (`() => …`), а не
строкой: символы `$` в CSS/JS ребёнка иначе истолковались бы как `$&`/`$1`.
`</script>` в JS по-прежнему экранируется (`<\/script>`).

Перед пользовательским JS внедряются два служебных скрипта: `CONSOLE_HOOK` (мост
консоли) и `LINK_GUARD` — перехватывает клики по якорным ссылкам (`#section`):
гасит переход (иначе в `srcdoc` он разрешается относительно адреса платформы и
грузит её в превью) и сам плавно прокручивает к секции с нужным id. Секции
магазина имеют id `home`/`products`/`contacts` (см. PITFALLS).

### localStorage
Один ключ `savedLessonCode` со значением `JSON.stringify({html,css,js})`.
Чтение обёрнуто в `try/catch` (повреждённый/устаревший JSON → ведём себя как «нет данных»).
Сброс — `localStorage.removeItem('savedLessonCode')`.

### Вкладки
4 вкладки: `task.md` (рендер Markdown в `.markdown-body`), `index.html`, `style.css`,
`main.js` (textarea). Переключение **только показывает/прячет** блоки — textarea
не пересоздаются, поэтому содержимое не теряется. Источник истины для кода — сами textarea
(+ зеркало в localStorage), а не переменные.

### Markdown-парсер
Поддержка: `# ## ###`, списки `-`/`*` (без вложенности), `**жирный**`, `*курсив*`,
`` `код` ``, абзацы (разделение пустыми строками). Порядок обработки важен —
см. PITFALLS (экранирование HTML, code → bold → italic, списки построчно).

### Редактор кода: подсветка синтаксиса + перехват Tab

Подсветка в `<textarea>` напрямую невозможна, поэтому используется классический
приём **«оверлей»**: внутри обёртки `.editor-wrap` (position: relative) лежат два
слоя с идентичными метриками шрифта/отступов:

```
.editor-wrap (relative, overflow:hidden)
├─ <pre class="highlight"><code>…</code></pre>   ← z-index 1, раскрашенный код
└─ <textarea class="editor">                     ← z-index 2, color:transparent,
                                                     caret-color виден, ввод сюда
```

- Textarea сверху, текст прозрачный (`color: transparent`), виден только курсор
  (`caret-color`) и выделение (`::selection` с полупрозрачным фоном — цветной код
  под ним просвечивает).
- При `input` пересобираем `<code>.innerHTML` через `highlight(value, lang)`.
- При `scroll` синхронизируем `pre.scrollTop/scrollLeft = textarea.scrollTop/Left`
  (`syncScroll`). У `pre` `overflow:hidden`, прокрутка только программная.

**Нумерация строк.** В обёртку добавлен третий слой `.gutter` (слева, ширина
`--gutter-w`, `z-index:3`, `pointer-events:none`). `updateGutter(editor)` пишет в
него номера строк (`textContent`, по числу `\n`) при каждой перерисовке подсветки.
Метрики (шрифт, `line-height`, верхний `padding`) совпадают со слоями кода →
номера вровень со строками. Оба слоя кода получают `padding-left: var(--gutter-w)+10px`,
освобождая место под колонку. `syncScroll` двигает gutter только по вертикали
(`scrollTop`), по горизонтали он стоит на месте (как в настоящих редакторах);
`paddingLeft` уже учтён в `MIRROR_PROPS`, поэтому Emmet-превью не смещается.

**Своя история отмены/повтора (переживает перезагрузку).** Нативная история
textarea стирается при перезагрузке (контент грузится через `value=`), поэтому
ведём собственный стек снимков на каждый редактор и храним его в localStorage
(ключ `undoHistory`). Снимок — `{v:значение, s,e:выделение}`. `histRecord(ta)`
вызывается из обработчика `input`: срезает «redo»-ветку, добавляет шаг (или
**сливает** быстрый набор в текущий шаг — коалесинг по времени `COALESCE_MS` и
неизменному числу строк), ограничивает длину `HIST_MAX`. Ctrl+Z / Ctrl+Shift+Z
(и Ctrl+Y) перехватываются в `keydown` ДО нативной отмены (`histUndo`/`histRedo`
→ `histApply`: ставит `value`+выделение под флагом `restoring`, затем
`updateHighlight`+`autosave`). Запись в localStorage дебаунсится
(`saveHistorySoon`) + флаш на `beforeunload`; при переполнении квоты стеки
ужимаются вдвое. На старте `histLoad` поднимает стек (и ставит значение по
указателю); `resetToTemplate` заводит историю заново с чистого шаблона.

**Перенос строк (Alt+Z, как в VS Code).** Глобальный флаг `wordWrap` +
`setWordWrap(on)` переключаются по Alt+Z (обработчик `keydown` редактора,
`e.code === "KeyZ"`). Включает класс `body.word-wrap`: оба слоя кода получают
`white-space:pre-wrap; overflow-wrap:anywhere; scrollbar-gutter:stable`, у
textarea `wrap="soft"`, горизонтальный скролл скрыт (`scrollbar-gutter:stable` на
обоих слоях — чтобы перенос textarea и видимого `<pre>` совпадал, несмотря на
вертикальный скроллбар).

**Нумерация при переносе сохраняется.** Логическая строка может занять несколько
визуальных: номер ставится у первой, а на строки-продолжения в `.gutter` идут
ПУСТЫЕ записи (номера остаются вровень с кодом). `updateGutter` в режиме переноса
вызывает `wrappedRowCounts(editor, lines)` — меряет «зеркалом» (по одному `div`
на строку, ширина = текстовая область слоя `<pre>`, `pre-wrap`+`anywhere`,
один пересчёт раскладки), сколько визуальных строк занимает каждая логическая.
Пересчёт при вводе, смене вкладки (`switchTab`), ресайзе колонок/окна
(`refreshWrapForVisible`). `caretCoords` в режиме переноса ставит зеркалу
`white-space:pre-wrap`+`overflow-wrap:anywhere` (ширину копирует), чтобы курсор,
цветные чипы и Emmet-превью совпадали с раскладкой. Настройка — в localStorage
(`wordWrap`), восстанавливается в `init`.

**Умный Enter (как в VS Code).** `smartEnter(editor)` в обработчике `keydown`
(plain Enter, без модификаторов, без выделения; иначе — обычный Enter):
сохраняет отступ текущей строки, углубляет его на уровень (`\t`) после
открывающего тега/`{`/`(`/`[`, а если курсор стоит МЕЖДУ парой «открыли—закрыли»
(`<p>|</p>` — для HTML через `endsWithOpenTag`/проверку `</`; `{|}`, `(|)`, `[|]`
— везде) раскрывает на три строки: открытие на месте, курсор на средней строке с
отступом на уровень глубже, закрытие на своей строке на исходном уровне. Правка
идёт через `insertAsUserInput` (Undo/история/подсветка/автосейв сами), затем
курсор ставится на среднюю строку.

**Цветовые чипы у hex-цветов (только в CSS-редакторе).** Четвёртый слой
`.swatch-layer` (`position:absolute; inset:0; overflow:hidden;
pointer-events:none; z-index:3`) — над textarea. `refreshSwatches(editor)`
(только при `data-lang="css"`) ищет `#`+hex-цифры (`COLOR_RE = /#[0-9a-fA-F]+/g`,
длину считаем сами) и рисует у каждого маленький цветной квадрат-`<button>`
(`.color-swatch`, `pointer-events:auto` — кликабелен, остальной слой клики
пропускает к textarea). **Чип показывается, пока введено 3–6 hex-цифр**
(`CHIP_MIN..CHIP_MAX`): при наборе 3-й цифры появляется, при 7-й — исчезает.
ID-селекторы отсеиваются `isColorHash` (у `#` в начале строки чип не рисуем).
Позиция — через «зеркало» (`caretCoords`): **слева от `#`, на пробелах** перед
ним (с вычетом `scrollTop/Left`; слой не скроллится, а пересобирается на
`input`/`scroll`/смене вкладки/развороте колонки; у скрытого редактора
`offsetParent===null` — пропускаем).

Чтобы квадрат не закрывал `:` и текст, `ensureColorSpacing(editor)` держит перед
`#hex` нужный зазор: **2 пробела, пока показан чип (3–6 цифр), иначе 1** — лишний
пробел появляется ВМЕСТЕ с чипом и убирается, когда чип исчезает (>6 цифр).
`applyGap` дописывает/удаляет пробелы через `execCommand('insertText'|'delete')`
(сохраняет Undo; повторный вход через `input` гасит флаг `adjustingSpacing`;
каретка восстанавливается с поправкой). Зовётся из обработчика `input` (перед
`updateHighlight`) и из `init` для загруженного CSS.

Клик по чипу открывает **скрытый нативный `<input type="color">`**
(`#color-input`, `.click()`), дающий палитру и пипетку ОС; по событию `change`
выбранный `#rrggbb` подставляется через `insertAsUserInput` (Ctrl+Z отменяет,
подсветка/автосейв обновляются сами). 3-значный hex нормализуется в 6-значный
(`normalizeHex`) — нативный input иначе не принимает. Чип не меняет ширину текста
(отдельный слой), поэтому метрики textarea/подсветки не «разъезжаются».

Подсветка — собственный токенайзер `tokenize(code, patterns)` (без библиотек):
бежим по строке, на каждой позиции пробуем **липкие** регэкспы (флаг `/y`) по
порядку; совпало — оборачиваем в `<span class="tok-…">` с `escapeHtml`, иначе
экранируем один символ. Наборы правил: `HTML_PATTERNS`, `CSS_PATTERNS`,
`JS_PATTERNS` (язык берётся из `data-lang` у textarea). Цвета — классы `.tok-*`
в `style.css`. Подсветка контекстно-свободная (упрощённая), для учебного кода
достаточно.

**Tab вставляет символ табуляции** (`\t`), а не уводит фокус: `keydown` →
`preventDefault()` → `insertTab()`. Внутри — `document.execCommand('insertText',
false, '\t')`: сохраняет нативную историю отмены (Ctrl+Z) и сам диспатчит `input`
(→ подсветка + автосохранение). Фолбэк на ручную вставку с эмуляцией `input`,
если `execCommand` недоступен. Размер таба — `tab-size: 4` (одинаково у обоих слоёв).

### Emmet (раскрытие HTML-аббревиатур) — свой движок, файл `emmet.js`

Настоящий Emmet — большая библиотека; тащить её (даже локально) против принципа
«без библиотек». Поэтому написан **свой мини-Emmet** (как и Markdown-парсер, ZIP,
подсветка). Файл `emmet.js` подключается в `index.html` ПЕРЕД `main.js`, экспортирует
`window.Emmet = { expand(abbr) → {text, caret} | null, isExpandable(abbr) }`.

Поддержка (подмножество): вложенность `>`, соседи `+`, подъём `^`/`^^`, повтор `*N`,
группы `(...)`, `#id`, `.class`, `[attr=val]`, текст `{...}`, нумерация `$`/`$$`/`$@-N`,
неявный `div`, void-теги, атрибуты по умолчанию (`a[href]`, `img[src][alt]` …).
Сниппеты: `!` и `html:5` раскрываются в скелет HTML5-страницы (через словарь
`SNIPPETS` в `emmet.js`, метка курсора `{{CARET}}` — внутри `<body>`); это не
разбираемые аббревиатуры, а готовые куски, обрабатываются в `expand`/`isExpandable`
до токенайзера. CSS-аббревиатуры (`m10`) НЕ поддерживаются — это отдельный большой
словарь, вне scope.

Пайплайн: `tokenize` (учитывает скобки) → `parseSequence` (дерево; рекурсия для групп,
указатель `parent` для `^`) → `renderNode/renderElement` (HTML с табами-отступами).
Курсор после раскрытия ставится во **первый пустой элемент** (метка-пробел `MARKER`,
вырезается на выходе, её индекс = смещение каретки).

Встройка в редактор (`main.js`): Tab → `tryExpandEmmet()` только для `data-lang="html"`.
`extractAbbreviation()` берёт аббревиатуру слева от каретки (учёт скобок; стоп на
пробеле/`\n`; если упёрлись в `<` — это готовая разметка, не трогаем). Раскрытие
вставляется через `execCommand('insertText')` (сохраняет Undo и шлёт `input` →
подсветка/автосохранение); каждая новая строка получает отступ текущей строки.
Если аббревиатуры нет/невалидна — обычная вставка `\t`. Shift+Tab — всегда таб.

**Превью у курсора (`updateEmmetPreview` в `main.js`).** Пока ребёнок печатает в
HTML-редакторе, всплывающее окно `#emmet-preview` (фикс. позиция, `pointer-events:
none`) показывает структуру, которую вставит Tab. Источник — те же
`extractAbbreviation`/`isExpandable`/`expand`, поэтому превью ровно совпадает с
будущей вставкой; код в окне раскрашивается тем же `highlight(text, 'html')`.
Пиксельные координаты курсора в `<textarea>` берём приёмом **«зеркало»**
(`caretCoords`): невидимый `div` с теми же метриками (`MIRROR_PROPS`), внутри —
текст до курсора + span-маркер, у маркера читаем `offsetLeft/offsetTop`. Окно
позиционируется под строкой курсора (с клампом в окно; если не влезает вниз —
показывается над строкой). Обновляется на `input`, `scroll`, перемещение каретки
(`keyup` со стрелками/Home/End, `click`); скрывается на `blur`, `Escape`, Tab
(раскрытие) и смене вкладки.

### Встроенная консоль (без DevTools)

Цель: ребёнок видит `console.log` и ошибки прямо на платформе. Решение — мост через
`postMessage`:

1. В собираемый документ iframe (`updateIframe`) ПЕРЕД пользовательским `<script>`
   вставляется `CONSOLE_HOOK` — он подменяет `console.log/info/warn/error/debug` и
   слушает `error` / `unhandledrejection`, пересылая всё родителю:
   `parent.postMessage({__console:true, level, text}, '*')`. Оригинальный `console`
   тоже вызывается (вывод дублируется в настоящий DevTools).
2. Платформа слушает `window 'message'` (`handleConsoleMessage`), принимает только от
   `preview.contentWindow`, и добавляет строки в `#console-output` (цвет по уровню,
   автоскролл, лимит 500 строк). Текст вставляется через `textContent` (без HTML-инъекций).
3. Консоль очищается при каждом «Запустить» (`updateIframe → clearConsole`) и кнопкой
   «Очистить». Объекты сериализуются `JSON.stringify` с `try/catch` (циклические → `String`).

Работает благодаря `allow-same-origin` + `allow-scripts` в sandbox (иначе `parent`/
`postMessage` из iframe были бы недоступны).

### Многошаговый урок «Собери свой интернет-магазин» (steps + кнопки-вставки)

Активный урок теперь — конструктор интернет-магазина. Механизм «вставь готовый
блок кнопкой → нажми Запустить → измени деталь сам».

**Модель данных.** Расширен объект `lesson` (не введён мульти-урочный механизм —
платформа однопользовательская, селектора уроков нет по решению выше). Поля:
`title`, `introMd`, `outroMd`, `hint` (общая), `initialHTML/CSS/JS` (пустые — сайт
«растёт» из блоков) и массив `steps[]`. Каждый шаг:
`{ title, time, goalMd, actionMd, snippets:{html?,css?,js?}, taskMd, hintMd, doneMd }`.
Активный урок один; старый «Измени цвет текста» остался в истории git.

**Рендер.** `renderLesson()` строит в панели `#task.md`: вступление → прогресс-бар
→ карточки шагов (`buildStepCard`) → финал (`.lesson-outro`, скрыт до 5/5).
Порядок частей карточки: заголовок → `goalMd` → `actionMd` → **инструкция
(`taskMd`) с кнопками внутри неё** → подсказка → `doneMd`. Текстовые части
рендерятся тем же `markdownToHtml`.

**Кнопки вставки идут ВНУТРИ инструкции (`appendTaskWithInlineButtons`).** В
`taskMd` ставятся метки-плейсхолдеры `[[btn:html]]` / `[[btn:css]]` / `[[btn:js]]`
там, где должна стоять кнопка соответствующей части: сначала кнопка нужной части,
сразу под ней — её кусок задания, потом метка следующей части и её кусок. Рендер
делит `taskMd` по этим меткам (`split` с группой), чередуя `.step-task`-блоки и
`.step-buttons` с одной кнопкой. Если меток в `taskMd` нет — откат к старому виду:
вся инструкция, под ней все кнопки разом. Кнопки по-прежнему `.btn-insert` с
`dataset.insert/lang`, поэтому `updateProgress` (общий `querySelectorAll`) и
порядок разблокировки html→css→js не зависят от их размещения. Ручной шаг рендерит
инструкцию, под ней — одну кнопку «✓ Я выполнил этот шаг» (`makeManualDoneButton`).
Словарь `PART_INFO` задаёт для языка целевой файл, подпись и редактор.

**Вставка.** `insertPart(i, lang)` вставляет ОДНУ часть блока в её файл, помечает
ключ `"i:lang"` в `doneParts`, зовёт `autosave` + `updateProgress` и показывает
тост (что вставлено и что дальше: оставшиеся части / «нажми ▶ Запустить»). Превью
НЕ перерисовывается автоматически — ребёнок жмёт «Запустить» сам. Детали вставки:

- Вставка идёт **как пользовательский ввод** (`insertAsUserInput`): платформа
  переключает вкладку на целевой файл (`switchTab` — скрытый textarea нельзя
  сфокусировать, а заодно ребёнок видит, куда лёг код), ставит выделение на место
  вставки (`setSelectionRange(start, end)`) и зовёт `execCommand('insertText')`.
  Поэтому **Ctrl+Z откатывает вставку одним шагом**, как свой ввод; событие
  `input` диспатчится самим execCommand → подсветка и автосейв по обычному пути.
  Фолбэк (execCommand недоступен) — ручная правка `value` + `input` (без Undo).
  После вставки редактор прокручивается к коду (`caretCoords` + `scrollTop`).
- Каждый блок оборачивается **комментариями-границами** (`wrapBlock`):
  `<!-- ===== НАЧАЛО: Название ===== -->` … `КОНЕЦ` (для CSS — `/* */`, для JS — `//`).
  Видно, где блок начинается и кончается.
- **HTML вставляется ПЕРЕД `</body>`**, с отступом на уровень глубже. Если
  `</body>` нет (ученик не создал каркас) — вставка отменяется и показывается
  тост-ошибка «Сначала создай каркас сайта» (Emmet «!»). Поэтому **Шаг 0
  урока** — создать каркас: в `index.html` напечатать `!` + Tab.
- **CSS и JS** — отдельные файлы, дописываются в конец: выделяются хвостовые
  пустые строки и заменяются на `\n\n` + блок (одним шагом отмены). У них нет
  `</body>`, требование каркаса к ним не относится.

**Линейная разблокировка (кросс-шаговая + ВНУТРИ шага).** Активны кнопки частей
только **первого недособранного шага** (`firstUndoneStep`; шаг готов, когда
вставлены все его части — `isStepDone`): это гарантирует правильный визуальный
порядок (шапка→баннер→…→вход→комментарии). **Внутри шага** части тоже
разблокируются по очереди `html → css → js` (`firstUndonePartOfStep`): пока не
вставлен HTML, кнопка CSS — «🔒»; пока нет CSS — «🔒» у JS. Так ребёнок видит
сырые `[ВПИШИ …]` на странице сразу после HTML, ещё до стилей. Кнопки будущих
шагов и не-первых частей текущего → «🔒» (disabled, в `title` — что вставить
сначала). Вставленная часть остаётся **кликабельной** → «↻ Вставить HTML снова»:
повторно вставляет сниппет (на случай, если ребёнок удалил его код — в т.ч. через
Ctrl+Z). Порядок применяется только к новым частям; повтор разрешён всегда.

**Ручные шаги (без сниппетов).** Шаг с `manual: true` (или вовсе без `snippets`)
— это шаг, где ребёнок печатает код **руками** (каркас через Emmet «!», первый
текст, первый CSS). У него нет кнопок вставки: `buildStepCard` рисует одну кнопку
**«✓ Я выполнил этот шаг»** (`markManualDone`). Готовность такого шага — единственная
псевдо-часть `"done"`: `stepLangs(step)` для ручного шага возвращает `["done"]`,
поэтому `isStepDone` ждёт ключ `"i:done"` в `doneParts`, а прогресс «N из M» и
линейная разблокировка работают без особых случаев. Гейтинг **мягкий**: факт
нажатия ▶ Запустить не отслеживается — «нажми Запустить» живёт в тексте задания.

**Название вкладки = `<title>` ученика (preview.html).** Браузер берёт имя вкладки
из `<title>` **самой** `preview.html`, а не из `<title>` внутри её `<iframe>`.
Поэтому `render()` достаёт `<title>…</title>` из строки документа ученика (regex)
и присваивает `document.title` (нет title → дефолт). Сценарий шага A1: поменял
`<title>` → ▶ Запустить → имя вкладки просмотра изменилось.

**Прогресс.** `updateProgress()` обновляет состояния всех кнопок, текст «N из M»
(N — полностью собранные шаги, `doneStepCount`), ширину `.progress-fill` и
видимость финала. Хранится в localStorage в том же ключе `savedLessonCode` —
поле `done:["i:lang", …]` рядом с `{html,css,js}`. Обратная совместимость: нет
`done` → пусто; **старый формат с числами** (индексы шагов) при загрузке
разворачивается во все части шага. `autosave()` — единая функция (код +
прогресс), вызывается из `input`-обработчика и из `insertPart`.

**Сброс.** «↺ Начать заново» (`resetToTemplate`) очищает редакторы и `doneParts`,
удаляет ключ, перерисовывает шаги (кнопки снова активны).

**Подтверждение сброса (модальное окно).** Чтобы случайный клик не стёр весь
прогресс, кнопка «Начать заново» больше НЕ зовёт `resetToTemplate` напрямую — она
открывает модальное окно `#reset-modal` (`openResetModal`). Сброс выполнится,
только если ребёнок **вручную введёт фразу** `RESET_CONFIRM_PHRASE` («Да, я уверен,
что хочу начать заново и знаю, что это необратимо») в текстовое поле: `input` →
`validateResetPhrase` снимает `disabled` с кнопки «Да, начать заново» лишь при
точном совпадении (с `trim`), `confirmReset` закрывает окно и зовёт
`resetToTemplate`. Закрыть без сброса — Escape, клик по затемнённому фону
(`e.target === overlay`) или «Отмена»; Enter в поле = подтверждение. Стили —
`.modal-overlay/.modal/...` и кнопка `.btn-danger` (`style.css`).

**Картинки товаров — эмодзи**, не внешние URL: соблюдает «только локально, без
CDN» и их весело менять в задании «сделай сам».

**Модель «заполни пропуски» (ключевой учебный приём).** Сниппеты вставляются НЕ
готовыми: вместо реального текста в них стоят **метки-пропуски** вида
`[ВПИШИ НАЗВАНИЕ МАГАЗИНА]`, `[ВПИШИ ЦЕНУ]` и т.п. Цикл ребёнка на каждом шаге:
1) вставить блок кнопками (HTML+CSS), 2) открыть код, 3) найти метки и **вписать
свой текст** (метка стирается целиком, вместе со скобками), 4) нажать «Запустить».
Это главная активность урока — именно она наполняет 1–1.5 часа реальной работой
руками (≈29 пропусков на 7 шагов). Проверка — **визуальная**: незаполненная метка
видна прямо на сайте (`[ВПИШИ …]` рендерится как обычный текст), поэтому ребёнок
сам замечает, где не дописал. Отдельного «движка проверки правильности» нет (вне
scope, ТЗ не требует).

**Подсветка меток.** Чтобы пропуски было легко находить, в токенайзер добавлен
паттерн `TODO_PATTERN` (`/\[ВПИШИ[^\]]*\]/y`, первый во всех наборах
HTML/CSS/JS) — метка красится классом `.tok-todo` (ярко-жёлтый фон). ВАЖНО: у
`.tok-todo` разрешены только `color`/`background`/`border-radius` — `padding`,
`border`, `font-weight` изменили бы ширину текста и «разъехали» бы слой подсветки
с textarea (см. PITFALLS про метрики слоёв).

1. **Каркас + title** (ручной, Emmet «!») · 2. **Первый текст руками** (ручной) ·

6. Товары (+ввод Emmet) · 7. О нас (+своё свойство) · 8. Отзывы (+Emmet) ·
9. Подвал · 10. Корзина (JS) · 11. **Личный кабинет/вход ⚠** (HTML→CSS→JS) ·
12. **Комментарии ⚠** (HTML→CSS→JS).

**Уязвимые блоки (намеренно, для урока 2 «Этичный хакер»).** Шаги 11–12 строятся
как обычные фичи, но содержат **заложенные дыры** — на следующем уроке ребёнок
ищет/чинит их на своём же сайте. Работает благодаря тому, что iframe превью имеет
`allow-same-origin` + `srcdoc` (origin = origin платформы), значит `localStorage`
сайта = `localStorage` платформы, а `innerHTML`/`alert` исполняются:
- **Вход (11):** пароль захардкожен в main.js (`admin`/`megapass123`) + «секрет» в
  HTML-комментарии; приветствие через **`innerHTML`** (XSS через имя); флаг
  `is_admin` в localStorage открывает «🛠 Админ-панель» с секретным кодом (повышение
  прав на клиенте); приветствие из `location.hash` через `innerHTML` (отражённый XSS).
- **Комментарии (12):** `comments.innerHTML += text` без экранирования + сохранение
  в localStorage → **stored XSS**.
Точные сценарии эксплуатации и починки (`innerHTML`→`textContent`, убрать секрет с
клиента) — в `docs/LESSON_REDESIGN.md` (урок 2).

### Многостраничный магазин урока 1 (псевдостраницы, аккаунты, товар-страница)

Урок 1 собирает **многостраничное** приложение в одном `srcdoc`-документе. Так
как редактор правит один HTML-файл, а механизм вставки (`insertHtmlBeforeBody`)
лишь дописывает HTML перед `</body>`, «страницы» сделаны **псевдостраницами**:
секции лежат плоскими соседями, а JS-роутер сайта показывает/прячет их группами.

- **Роутер.** В коде сайта (в `main.js` ученика, вставляется шагом «Навигация»)
  есть карта `PAGES` (имя страницы → список id секций) и `showPage(name)`:
  прячет все секции из `ALL_SECTIONS` (`style.display="none"`) и показывает группу
  страницы. Шапка и подвал — вне карты, видны всегда. Стартовый `showPage("home")`.
- **Навигация без правок платформы.** Ссылки меню — `<a href="#" data-link="home">`,
  карточки — `<a class="card-link" data-product="N">`. Один делегированный
  `document`-обработчик кликов в коде сайта читает `data-link`/`data-product` и
  зовёт `showPage`/`openProduct`. **Конфликта с `LINK_GUARD` нет:** на `href="#"`
  страж гасит переход (id пустой → скролла нет), а пользовательский обработчик
  (фаза всплытия) всё равно срабатывает и переключает страницу. Поэтому
  `js/preview.js` НЕ менялся. Хеш-роутинг сознательно НЕ используется (страж
  гасит `preventDefault`-ом изменение `location.hash`).
- **Страница товара — data-driven.** Одна скрытая секция `#product-detail`
  заполняется `openProduct(id)` из массива `products` (id→{эмодзи,название,цена,
  описание}) через `textContent` (поля товара статичны — безопасно). Внутри —
  блок комментариев этого товара.
- **Аккаунты (НАМЕРЕННО уязвимо).** Регистрация дописывает `{login, password}` в
  массив `users` в localStorage (пароли открытым текстом). Вход сверяет с `users`
  **и** хранит backdoor `admin`/`megapass123`; `is_admin === "true"` открывает
  админ-панель с секретом. `currentUser`/`isLoggedIn` — в localStorage. Логин в
  правом верхнем углу — `#user-box`, обновляется `renderUserBox` через `innerHTML`
  (XSS через имя). Приветствие и отражённый XSS из `location.hash` — тоже `innerHTML`.
- **Комментарии по товару с гейтом по входу.** Ключ `comments_<id>` в localStorage,
  массив `{author, text}`. `renderCommentForm(id)` показывает форму только если
  `isLoggedIn === "true"`, иначе приглашение войти (ссылка `data-link="login"`).
  При отправке автор = `currentUser`. Рендер через `innerHTML` (stored XSS + XSS
  через имя автора) — для урока 2.
- **Защита от неполной сборки.** Все кросс-блочные вызовы между шагами защищены
  (`typeof renderUserBox === "function"`, `typeof openProduct === "function"`,
  `if (el) …`): пока ребёнок не вставил соответствующий блок, сайт не падает.
- **Согласование с уроком 2.** `index2.html` использует свой одностраничный
  `VULN_*`-сайт и после этой переделки урока 1 ему больше не соответствует —
  обновление вынесено в отдельную задачу (см. `docs/PLAN.md`).

**Чит для разработчика (`js/dev-cheat.js`).** Чтобы не пересобирать магазин по
шагам при разработке/тесте, есть «чит»: `loadDemoShop()` загружает готовый
собранный сайт в три редактора, отмечает все шаги выполненными (`doneParts` ×
`stepLangs`), сохраняет (`autosave`), обновляет прогресс и запускает превью.

Способы вызова: **(1) секретный адрес** `index.html#mega-admin` (или
`?mega-admin`) — `checkSecretUnlock` сохраняет флаг `megaDevAdmin="true"` в
localStorage и чистит адрес (`history.replaceState`); после этого `injectDevButton`
на КАЖДОЙ загрузке добавляет в `.control-buttons` постоянную кнопку
**«🛠 Весь магазин»** (заходить по секрету повторно не нужно). **(2)** Ctrl+Alt+D.
**(3)** `loadDemoShop()` в консоли (DevTools может попросить ввести «allow
pasting»). Кнопка — самый удобный путь (не блокируется DevTools). `DEV_HTML/CSS/JS` — это в точности результат
прохождения урока (все метки `[ВПИШИ …]` заполнены, добавлены Emmet-элементы:
5-я карточка, 4-й отзыв; подвал размещён последним для корректной раскладки
подстраниц). Сверка `DEV_JS === собранный JS урока` держится проверкой (чит не
«дрейфует» от сниппетов). Уязвимости сохранены — это тестовый сайт для урока 2.
Модуль подключён только в `index.html`.

### Два урока на одной платформе (урок 2 «Этичный хакер»)

Платформа однопользовательская, селектора уроков нет. Урок 2 сделан **отдельной
точкой входа `index2.html`** (решение пользователя 2026-06-14): тот же набор
модулей `js/`/`css/`, но вместо `js/lesson-data.js` подключён

платформы (рендер шагов, редактор, превью, ZIP…) **не дублируется и не меняется**.

- **Стартовый код урока 2 — вшитый эталонный уязвимый сайт** (`VULN_HTML/CSS/JS`
  в `lesson-data-security.js`): полный собранный «МегаМагазин» с заложенными
  дырами. Одинаков у всех детей → не зависит от того, что реально доделано в

- **Шаги урока 2 — все `manual:true`** (S0–S5): ребёнок исследует/эксплуатирует и
  **правит код в редакторах** по циклу 🔴 найди → 🟢 почини (▶ Запустить) → ✅ проверь,

- **Раздельное хранилище.** Чтобы прогрессы уроков не затирали друг друга,

## Отклонения от исходного ТЗ (deepseek) и обоснования

1. **`srcdoc` вместо `document.write`.** document.write в iframe требует доступа к
   `contentDocument` и при пустом `src` даёт нестабильное поведение/гонки; `srcdoc`
   декларативен и дружит с `sandbox`.
2. **Экранирование HTML в `markdownToHtml`.** В ТЗ не упомянуто, но без него текст
   вроде `<p>` или `&` в описании сломает разметку. Экранируем `& < >` до парсинга.
3. **Явный MIME у `downloadFile`.** Передаём третий аргумент (`text/html`, `text/css`,
   `application/javascript`), а не угадываем по расширению.
4. **`try/catch` вокруг чтения localStorage.** Защита от повреждённых данных и
   режимов приватности, где доступ к storage кидает исключение.
5. **iframe обновляется только по «Запустить».** В ТЗ это так и задумано (ручной
   запуск). На `input` — только автосохранение, без перерисовки превью.
6. **Добавлен `allow-modals` в `sandbox`.** Без него `alert/confirm/prompt` из
   кода ребёнка молча блокируются. ТЗ этот флаг не указывал.

## Чего НЕ делаем (вне scope)

- Нет создания/выбора уроков, нет роутинга, нет бэкенда.
- Нет внешних библиотек, шрифтов с CDN, сборщиков.
- Нет автоматической проверки «правильно ли решена задача» (ТЗ не требует).
