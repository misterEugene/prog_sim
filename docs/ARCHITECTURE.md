# Архитектура и решения по реализации

Документ описывает целевую архитектуру платформы, ключевые решения и обоснования.
Дополняй его, когда принимаешь новое нетривиальное решение.

## Концепция

Локальная (без сервера/сборки/библиотек) платформа для обучения детей коду.
**Один** предустановленный урок, зашитый в `main.js` (объект `lesson`).
Создание новых уроков через UI **не предусмотрено** — это намеренное упрощение.

Ребёнок: читает задачу (`task.md`), правит HTML/CSS/JS, жмёт «Запустить», видит
результат в iframe. Прогресс автосохраняется в `localStorage`, восстанавливается
после перезагрузки. Есть сброс к шаблону, подсказка (тост) и скачивание файлов.

## Данные урока (источник истины — объект `lesson` в main.js)

| Поле            | Значение |
|-----------------|----------|
| `title`         | «Измени цвет текста» (в UI: «Урок 1: Измени цвет текста») |
| `descriptionMd` | Markdown-описание (заголовок, задача, подсказка про `color`) |
| `initialHTML`   | `<p>Привет, я учусь программировать!</p>` |
| `initialCSS`    | `''` (пусто) |
| `initialJS`     | `''` (пусто) |
| `hint`          | «Напиши в CSS: p { color: blue; }» |

## Функции main.js и связи между ними

- `markdownToHtml(md)` — мини-парсер Markdown (без библиотек).
- `renderDescription()` — кладёт `markdownToHtml(lesson.descriptionMd)` в `.markdown-body`.
- `loadFromLocalStorage()` → `{html,css,js}` | `null`.
- `saveToLocalStorage(html,css,js)` — пишет один ключ `savedLessonCode` (JSON).
- `buildDocument()` — собирает полный HTML-документ из текущих редакторов (стили +
  служебные скрипты). Общий для превью и вкладки просмотра.
- `updateIframe()` — `clearConsole()` + кладёт `buildDocument()` в `iframe.srcdoc`
  и зеркалит документ в localStorage (`savePreviewDoc`, ключ `previewDoc`).
- `savePreviewDoc(doc)` — пишет готовый документ в localStorage (try/catch).
- `openInNewTab()` — сохраняет свежий снимок и открывает `preview.html` в
  именованной вкладке (`window.open(..., 'shopPreview')`). Если popup заблокирован —
  тост с подсказкой.

### Отдельная вкладка просмотра — `preview.html` (живое зеркало через localStorage)

Кнопка «🔗 Открыть в новой вкладке» открывает **отдельный файл `preview.html`**, а
НЕ Blob-снимок. Почему: Blob-URL — статичен (не отражает правки) и при отзыве
(`revokeObjectURL`) вкладка «ломалась» после перезагрузки.

Механизм: на каждый «Запустить» платформа кладёт готовый документ в localStorage
(`previewDoc`). `preview.html` содержит полноэкранный `<iframe>`, читает `previewDoc`
в свой `srcdoc` и подписан на событие **`storage`** (приходит из другой вкладки того
же origin) → переотображается синхронно с «Запустить». Плюсы: отражает изменения
(по Run, как и встроенное превью), переживает перезагрузку вкладки, ничего не
«отзывается». Нет дублирования логики сборки — `preview.html` лишь показывает уже
собранный платформой документ. (Событие `storage` стабильно по HTTP; на `file://`
зависит от браузера, но первичная загрузка/перезагрузка работают.)
- `resetToTemplate()` — заполняет редакторы `initial*`, чистит localStorage, зовёт `updateIframe()`.
- `crc32(bytes)` / `buildZip(files)` — ручная сборка ZIP-архива (метод store, без библиотек).
- `downloadBlob(blob, filename)` — `<a download>` + клик + revokeObjectURL.
- `downloadProject()` — собирает три файла в ZIP и скачивает `project.zip`.
- `showHint()` — показывает тост `lesson.hint`, прячет через 5 с.
- `switchTab(tabId)` — переключает активную вкладку и видимый блок.

Поток данных:
```
input в textarea ──> saveToLocalStorage()         (iframe НЕ трогаем)
кнопка «Запустить» ──> updateIframe()             (читает текущие textarea)
загрузка страницы ──> loadFromLocalStorage() || initial* ──> renderDescription() ──> updateIframe()
```

## Ключевые решения

### Сборка документа для iframe
`updateIframe()` присваивает результат `iframe.srcdoc` (**`srcdoc`, а не
`document.write`** — см. PITFALLS). Две ветки:

- **Фрагмент** (в HTML-редакторе нет `</body>`): оборачиваем как раньше —
  `<!DOCTYPE html><html><head><style>CSS</style></head><body>HTML+scripts</body></html>`.
- **Полный документ** (ученик создал каркас через Emmet «!», есть `</body>`):
  используем его как есть, лишь внедряя `<style>CSS</style>` перед `</head>` и
  скрипты (CONSOLE_HOOK + JS) перед `</body>`. Иначе вышел бы документ, вложенный
  в `<body>` (двойные `<html>/<head>/<body>`).

Замены `</head>`/`</body>` делаются **функцией-replacer** (`() => …`), а не
строкой: символы `$` в CSS/JS ребёнка иначе истолковались бы как `$&`/`$1`.
`</script>` в JS по-прежнему экранируется (`<\/script>`).

Перед пользовательским JS внедряются два служебных скрипта: `CONSOLE_HOOK` (мост
консоли) и `LINK_GUARD` — перехватывает клики по якорным ссылкам (`#section`):
гасит переход (иначе в `srcdoc` он разрешается относительно адреса платформы и
грузит её в превью) и сам плавно прокручивает к секции с нужным id. Секции
магазина имеют id `home`/`products`/`contacts` (см. PITFALLS).

### localStorage
Один ключ `savedLessonCode` со значением `JSON.stringify({html,css,js})`.
Чтение обёрнуто в `try/catch` (повреждённый/устаревший JSON → ведём себя как «нет данных»).
Сброс — `localStorage.removeItem('savedLessonCode')`.

### Вкладки
4 вкладки: `task.md` (рендер Markdown в `.markdown-body`), `index.html`, `style.css`,
`main.js` (textarea). Переключение **только показывает/прячет** блоки — textarea
не пересоздаются, поэтому содержимое не теряется. Источник истины для кода — сами textarea
(+ зеркало в localStorage), а не переменные.

### Markdown-парсер
Поддержка: `# ## ###`, списки `-`/`*` (без вложенности), `**жирный**`, `*курсив*`,
`` `код` ``, абзацы (разделение пустыми строками). Порядок обработки важен —
см. PITFALLS (экранирование HTML, code → bold → italic, списки построчно).

### Редактор кода: подсветка синтаксиса + перехват Tab

Подсветка в `<textarea>` напрямую невозможна, поэтому используется классический
приём **«оверлей»**: внутри обёртки `.editor-wrap` (position: relative) лежат два
слоя с идентичными метриками шрифта/отступов:

```
.editor-wrap (relative, overflow:hidden)
├─ <pre class="highlight"><code>…</code></pre>   ← z-index 1, раскрашенный код
└─ <textarea class="editor">                     ← z-index 2, color:transparent,
                                                     caret-color виден, ввод сюда
```

- Textarea сверху, текст прозрачный (`color: transparent`), виден только курсор
  (`caret-color`) и выделение (`::selection` с полупрозрачным фоном — цветной код
  под ним просвечивает).
- При `input` пересобираем `<code>.innerHTML` через `highlight(value, lang)`.
- При `scroll` синхронизируем `pre.scrollTop/scrollLeft = textarea.scrollTop/Left`
  (`syncScroll`). У `pre` `overflow:hidden`, прокрутка только программная.

Подсветка — собственный токенайзер `tokenize(code, patterns)` (без библиотек):
бежим по строке, на каждой позиции пробуем **липкие** регэкспы (флаг `/y`) по
порядку; совпало — оборачиваем в `<span class="tok-…">` с `escapeHtml`, иначе
экранируем один символ. Наборы правил: `HTML_PATTERNS`, `CSS_PATTERNS`,
`JS_PATTERNS` (язык берётся из `data-lang` у textarea). Цвета — классы `.tok-*`
в `style.css`. Подсветка контекстно-свободная (упрощённая), для учебного кода
достаточно.

**Tab вставляет символ табуляции** (`\t`), а не уводит фокус: `keydown` →
`preventDefault()` → `insertTab()`. Внутри — `document.execCommand('insertText',
false, '\t')`: сохраняет нативную историю отмены (Ctrl+Z) и сам диспатчит `input`
(→ подсветка + автосохранение). Фолбэк на ручную вставку с эмуляцией `input`,
если `execCommand` недоступен. Размер таба — `tab-size: 4` (одинаково у обоих слоёв).

### Emmet (раскрытие HTML-аббревиатур) — свой движок, файл `emmet.js`

Настоящий Emmet — большая библиотека; тащить её (даже локально) против принципа
«без библиотек». Поэтому написан **свой мини-Emmet** (как и Markdown-парсер, ZIP,
подсветка). Файл `emmet.js` подключается в `index.html` ПЕРЕД `main.js`, экспортирует
`window.Emmet = { expand(abbr) → {text, caret} | null, isExpandable(abbr) }`.

Поддержка (подмножество): вложенность `>`, соседи `+`, подъём `^`/`^^`, повтор `*N`,
группы `(...)`, `#id`, `.class`, `[attr=val]`, текст `{...}`, нумерация `$`/`$$`/`$@-N`,
неявный `div`, void-теги, атрибуты по умолчанию (`a[href]`, `img[src][alt]` …).
Сниппеты: `!` и `html:5` раскрываются в скелет HTML5-страницы (через словарь
`SNIPPETS` в `emmet.js`, метка курсора `{{CARET}}` — внутри `<body>`); это не
разбираемые аббревиатуры, а готовые куски, обрабатываются в `expand`/`isExpandable`
до токенайзера. CSS-аббревиатуры (`m10`) НЕ поддерживаются — это отдельный большой
словарь, вне scope.

Пайплайн: `tokenize` (учитывает скобки) → `parseSequence` (дерево; рекурсия для групп,
указатель `parent` для `^`) → `renderNode/renderElement` (HTML с табами-отступами).
Курсор после раскрытия ставится во **первый пустой элемент** (метка-пробел `MARKER`,
вырезается на выходе, её индекс = смещение каретки).

Встройка в редактор (`main.js`): Tab → `tryExpandEmmet()` только для `data-lang="html"`.
`extractAbbreviation()` берёт аббревиатуру слева от каретки (учёт скобок; стоп на
пробеле/`\n`; если упёрлись в `<` — это готовая разметка, не трогаем). Раскрытие
вставляется через `execCommand('insertText')` (сохраняет Undo и шлёт `input` →
подсветка/автосохранение); каждая новая строка получает отступ текущей строки.
Если аббревиатуры нет/невалидна — обычная вставка `\t`. Shift+Tab — всегда таб.

**Превью у курсора (`updateEmmetPreview` в `main.js`).** Пока ребёнок печатает в
HTML-редакторе, всплывающее окно `#emmet-preview` (фикс. позиция, `pointer-events:
none`) показывает структуру, которую вставит Tab. Источник — те же
`extractAbbreviation`/`isExpandable`/`expand`, поэтому превью ровно совпадает с
будущей вставкой; код в окне раскрашивается тем же `highlight(text, 'html')`.
Пиксельные координаты курсора в `<textarea>` берём приёмом **«зеркало»**
(`caretCoords`): невидимый `div` с теми же метриками (`MIRROR_PROPS`), внутри —
текст до курсора + span-маркер, у маркера читаем `offsetLeft/offsetTop`. Окно
позиционируется под строкой курсора (с клампом в окно; если не влезает вниз —
показывается над строкой). Обновляется на `input`, `scroll`, перемещение каретки
(`keyup` со стрелками/Home/End, `click`); скрывается на `blur`, `Escape`, Tab
(раскрытие) и смене вкладки.

### Встроенная консоль (без DevTools)

Цель: ребёнок видит `console.log` и ошибки прямо на платформе. Решение — мост через
`postMessage`:

1. В собираемый документ iframe (`updateIframe`) ПЕРЕД пользовательским `<script>`
   вставляется `CONSOLE_HOOK` — он подменяет `console.log/info/warn/error/debug` и
   слушает `error` / `unhandledrejection`, пересылая всё родителю:
   `parent.postMessage({__console:true, level, text}, '*')`. Оригинальный `console`
   тоже вызывается (вывод дублируется в настоящий DevTools).
2. Платформа слушает `window 'message'` (`handleConsoleMessage`), принимает только от
   `preview.contentWindow`, и добавляет строки в `#console-output` (цвет по уровню,
   автоскролл, лимит 500 строк). Текст вставляется через `textContent` (без HTML-инъекций).
3. Консоль очищается при каждом «Запустить» (`updateIframe → clearConsole`) и кнопкой
   «Очистить». Объекты сериализуются `JSON.stringify` с `try/catch` (циклические → `String`).

Работает благодаря `allow-same-origin` + `allow-scripts` в sandbox (иначе `parent`/
`postMessage` из iframe были бы недоступны).

### Многошаговый урок «Собери свой интернет-магазин» (steps + кнопки-вставки)

Активный урок теперь — конструктор интернет-магазина. Механизм «вставь готовый
блок кнопкой → нажми Запустить → измени деталь сам».

**Модель данных.** Расширен объект `lesson` (не введён мульти-урочный механизм —
платформа однопользовательская, селектора уроков нет по решению выше). Поля:
`title`, `introMd`, `outroMd`, `hint` (общая), `initialHTML/CSS/JS` (пустые — сайт
«растёт» из блоков) и массив `steps[]`. Каждый шаг:
`{ title, time, goalMd, actionMd, snippets:{html?,css?,js?}, taskMd, hintMd, doneMd }`.
Активный урок один; старый «Измени цвет текста» остался в истории git.

**Рендер.** `renderLesson()` строит в панели `#task.md`: вступление → прогресс-бар
→ карточки шагов (`buildStepCard`) → финал (`.lesson-outro`, скрыт до 5/5).
Текстовые части шага рендерятся тем же `markdownToHtml`. В карточке — **отдельная
кнопка на каждую часть блока** (ряд `.step-buttons`): по одной на каждый язык из
`snippets` (HTML / CSS / JS); обработчик `insertPart(i, lang)` вешается при
создании. Словарь `PART_INFO` задаёт для языка целевой файл, подпись и редактор.

**Вставка.** `insertPart(i, lang)` вставляет ОДНУ часть блока в её файл, помечает
ключ `"i:lang"` в `doneParts`, зовёт `autosave` + `updateProgress` и показывает
тост (что вставлено и что дальше: оставшиеся части / «нажми ▶ Запустить»). Превью
НЕ перерисовывается автоматически — ребёнок жмёт «Запустить» сам. Детали вставки:

- Вставка идёт **как пользовательский ввод** (`insertAsUserInput`): платформа
  переключает вкладку на целевой файл (`switchTab` — скрытый textarea нельзя
  сфокусировать, а заодно ребёнок видит, куда лёг код), ставит выделение на место
  вставки (`setSelectionRange(start, end)`) и зовёт `execCommand('insertText')`.
  Поэтому **Ctrl+Z откатывает вставку одним шагом**, как свой ввод; событие
  `input` диспатчится самим execCommand → подсветка и автосейв по обычному пути.
  Фолбэк (execCommand недоступен) — ручная правка `value` + `input` (без Undo).
  После вставки редактор прокручивается к коду (`caretCoords` + `scrollTop`).
- Каждый блок оборачивается **комментариями-границами** (`wrapBlock`):
  `<!-- ===== НАЧАЛО: Название ===== -->` … `КОНЕЦ` (для CSS — `/* */`, для JS — `//`).
  Видно, где блок начинается и кончается.
- **HTML вставляется ПЕРЕД `</body>`**, с отступом на уровень глубже. Если
  `</body>` нет (ученик не создал каркас) — вставка отменяется и показывается
  тост-ошибка «Сначала создай каркас сайта» (Emmet «!»). Поэтому **Шаг 0
  урока** — создать каркас: в `index.html` напечатать `!` + Tab.
- **CSS и JS** — отдельные файлы, дописываются в конец: выделяются хвостовые
  пустые строки и заменяются на `\n\n` + блок (одним шагом отмены). У них нет
  `</body>`, требование каркаса к ним не относится.

**Линейная разблокировка.** Активны кнопки частей только **первого недособранного
шага** (`firstUndoneStep`; шаг готов, когда вставлены все его части —
`isStepDone`): это гарантирует правильный визуальный порядок (шапка→баннер→
товары→подвал→JS). Внутри шага части можно вставлять в любом порядке (CSS/JS —
отдельные файлы, а HTML-части в разных шагах). Кнопки будущих шагов → «🔒»
(disabled, в `title` — какой блок собрать сначала). Вставленная часть остаётся
**кликабельной** → «↻ Вставить HTML снова»: повторно вставляет сниппет (на
случай, если ребёнок удалил его код — в т.ч. через Ctrl+Z). Порядок применяется
только к новым частям; повтор разрешён всегда.

**Прогресс.** `updateProgress()` обновляет состояния всех кнопок, текст «N из M»
(N — полностью собранные шаги, `doneStepCount`), ширину `.progress-fill` и
видимость финала. Хранится в localStorage в том же ключе `savedLessonCode` —
поле `done:["i:lang", …]` рядом с `{html,css,js}`. Обратная совместимость: нет
`done` → пусто; **старый формат с числами** (индексы шагов) при загрузке
разворачивается во все части шага. `autosave()` — единая функция (код +
прогресс), вызывается из `input`-обработчика и из `insertPart`.

**Сброс.** «↺ Начать заново» (`resetToTemplate`) очищает редакторы и `doneParts`,
удаляет ключ, перерисовывает шаги (кнопки снова активны).

**Картинки товаров — эмодзи**, не внешние URL: соблюдает «только локально, без
CDN» и их весело менять в задании «сделай сам».

**Модель «заполни пропуски» (ключевой учебный приём).** Сниппеты вставляются НЕ
готовыми: вместо реального текста в них стоят **метки-пропуски** вида
`[ВПИШИ НАЗВАНИЕ МАГАЗИНА]`, `[ВПИШИ ЦЕНУ]` и т.п. Цикл ребёнка на каждом шаге:
1) вставить блок кнопками (HTML+CSS), 2) открыть код, 3) найти метки и **вписать
свой текст** (метка стирается целиком, вместе со скобками), 4) нажать «Запустить».
Это главная активность урока — именно она наполняет 1–1.5 часа реальной работой
руками (≈29 пропусков на 7 шагов). Проверка — **визуальная**: незаполненная метка
видна прямо на сайте (`[ВПИШИ …]` рендерится как обычный текст), поэтому ребёнок
сам замечает, где не дописал. Отдельного «движка проверки правильности» нет (вне
scope, ТЗ не требует).

**Подсветка меток.** Чтобы пропуски было легко находить, в токенайзер добавлен
паттерн `TODO_PATTERN` (`/\[ВПИШИ[^\]]*\]/y`, первый во всех наборах
HTML/CSS/JS) — метка красится классом `.tok-todo` (ярко-жёлтый фон). ВАЖНО: у
`.tok-todo` разрешены только `color`/`background`/`border-radius` — `padding`,
`border`, `font-weight` изменили бы ширину текста и «разъехали» бы слой подсветки
с textarea (см. PITFALLS про метрики слоёв).

**Состав урока (7 шагов):** Шаг 0 (каркас Emmet «!») → 1. Шапка (название) →
2. Баннер (заголовок/подзаголовок/кнопка) → 3. Товары (4 карточки × эмодзи/
название/цена) → 4. Блок «О нас» (заголовок + рассказ) → 5. Отзывы (3 × текст/имя)
→ 6. Подвал (телефон/почта/адрес/копирайт) → 7. Корзина на JS (сообщение). CSS
вставляется готовым (дети не пишут стили), пропуски — только в наполняемом тексте.

**Тост.** `showHint` обобщён в `showToast(title, text)`; `insertBlock` использует
его для подсказки «блок добавлен».

## Отклонения от исходного ТЗ (deepseek) и обоснования

1. **`srcdoc` вместо `document.write`.** document.write в iframe требует доступа к
   `contentDocument` и при пустом `src` даёт нестабильное поведение/гонки; `srcdoc`
   декларативен и дружит с `sandbox`.
2. **Экранирование HTML в `markdownToHtml`.** В ТЗ не упомянуто, но без него текст
   вроде `<p>` или `&` в описании сломает разметку. Экранируем `& < >` до парсинга.
3. **Явный MIME у `downloadFile`.** Передаём третий аргумент (`text/html`, `text/css`,
   `application/javascript`), а не угадываем по расширению.
4. **`try/catch` вокруг чтения localStorage.** Защита от повреждённых данных и
   режимов приватности, где доступ к storage кидает исключение.
5. **iframe обновляется только по «Запустить».** В ТЗ это так и задумано (ручной
   запуск). На `input` — только автосохранение, без перерисовки превью.
6. **Добавлен `allow-modals` в `sandbox`.** Без него `alert/confirm/prompt` из
   кода ребёнка молча блокируются. ТЗ этот флаг не указывал.

## Чего НЕ делаем (вне scope)

- Нет создания/выбора уроков, нет роутинга, нет бэкенда.
- Нет внешних библиотек, шрифтов с CDN, сборщиков.
- Нет автоматической проверки «правильно ли решена задача» (ТЗ не требует).
