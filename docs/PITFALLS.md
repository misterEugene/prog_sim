# Частые ошибки и решения

Журнал граблей, чтобы не наступать на них повторно в новых сессиях.
Формат записи: **Проблема → Причина → Решение**. Добавляй новые сверху раздела.

## Markdown-парсер

### `<` `>` `&` из описания ломают разметку
- **Причина:** текст вставляется через `innerHTML`; необработанные символы трактуются как теги/сущности.
- **Решение:** экранировать `& < >` (`&amp; &lt; &gt;`) **в самом начале**, до применения любых правил Markdown.

### `*курсив*` конфликтует со списками `* пункт` и с `**жирным**`
- **Причина:** один символ `*` используется и для списков, и для курсива; жадный регэксп курсива «съедает» звёздочки жирного.
- **Решение:** обрабатывать построчно. Сначала строки-заголовки (`#`), затем строки-списки
  (строка начинается с `- ` или `* `). Инлайн-замены в порядке: `` `код` `` → `**жирный**`
  → `*курсив*`. Для жирного и курсива использовать нежадные регэкспы (`\*\*(.+?)\*\*`, затем `\*(.+?)\*`).

### Списки рендерятся как отдельные `<ul>` на каждый пункт
- **Причина:** наивная построчная замена оборачивает каждый `- item` в свой `<ul>`.
- **Решение:** группировать подряд идущие строки-пункты в один `<ul>`; закрывать `<ul>`,
  когда встретилась не-списковая строка.

## iframe / превью

### `document.write` даёт пустой/мигающий iframe или гонку
- **Причина:** доступ к `iframe.contentDocument` при пустом `src` ненадёжен; `open/write/close` конфликтует с sandbox.
- **Решение:** использовать `iframe.srcdoc = fullHtml`. Декларативно, без обращения к contentDocument.

### Скрипт пользователя не выполняется в iframe
- **Причина:** в `sandbox` нет `allow-scripts`.
- **Решение:** атрибут iframe должен содержать `allow-scripts`.

### `alert()` / `confirm()` / `prompt()` не срабатывают (скрипт «как будто не работает»)
- **Причина:** модальные диалоги в песочнице блокируются без флага `allow-modals`.
  Скрипт выполняется, но вызов молча игнорируется (в консоли браузера — предупреждение
  «Ignored call to 'alert()'. ...the 'allow-modals' keyword is not set.»). В исходном
  ТЗ этот флаг забыли.
- **Решение:** добавить `allow-modals` в `sandbox`. Итоговый набор:
  `sandbox="allow-same-origin allow-scripts allow-popups allow-forms allow-modals"`.

### Клик по якорной ссылке `#section` в превью загружает всю платформу в iframe
- **Причина:** документ `srcdoc` имеет URL `about:srcdoc`, но относительные и
  якорные ссылки (`#`, `#section`, `page.html`) разрешаются ОТНОСИТЕЛЬНО адреса
  родительской страницы (платформы). Поэтому `href="#products"` → `…/index.html#products`
  → это ДРУГОЙ документ → переход уводит iframe на саму платформу (после
  «Запустить»/перезагрузки `srcdoc` пересобирается — превью восстанавливается).
  Обычная якорная навигация внутри `srcdoc` из-за этого не работает в принципе.
- **Решение:** в собираемый документ добавить скрипт-«страж» `LINK_GUARD`: на
  `click` (capture) находит ближайший `<a>`; для `#`-ссылок гасит переход
  (`preventDefault`) и сам прокручивает к элементу с нужным id
  (`getElementById(href.slice(1)).scrollIntoView({behavior:'smooth'})`). Так
  якоря работают как ожидается, не уводя превью. Настоящие (внешние) ссылки не
  трогаем; top-навигацию sandbox и так блокирует (нет `allow-top-navigation`).
- Секциям магазина проставлены id (`home` — шапка, `products` — товары,
  `contacts` — подвал), пункты меню ссылаются на них.

### `</script>` в пользовательском JS рвёт документ iframe
- **Причина:** строка `</script>` внутри инлайнового `<script>` закрывает тег раньше времени.
- **Решение:** при сборке документа экранировать `</script>` → `<\/script>` в коде JS
  (и при желании в HTML), либо помнить про это ограничение. (Низкий приоритет для урока про цвет, но фиксируем.)

## Редактор: подсветка (оверлей) и Tab

### Текст ввода и подсветка «разъезжаются»
- **Причина:** у слоёв `.editor` (textarea) и `.highlight` (pre) разные метрики —
  шрифт, размер, `line-height`, `padding`, `tab-size`, рамка или `box-sizing`.
- **Решение:** задать ОБОИМ слоям одинаковые `font-family/size`, `line-height`,
  `padding`, `border:0`, `tab-size` (+ `-moz-tab-size`), `white-space: pre`,
  `letter-spacing: 0`, `box-sizing: border-box`. Оба `position:absolute; inset:0`.

### При прокрутке кода подсветка стоит на месте
- **Причина:** прокручивается только верхний `textarea`; нижний `pre` сам не двигается.
- **Решение:** на событие `scroll` у textarea синхронизировать
  `pre.scrollTop/scrollLeft = textarea.scrollTop/scrollLeft`. У `pre` `overflow:hidden`,
  `pointer-events:none` (чтобы клики шли к textarea).

### Подсветка ломает разметку / показывает теги как текст
- **Причина:** токены вставляются в `<code>` через `innerHTML`; неэкранированные
  `< > &` из кода ребёнка трактуются как теги.
- **Решение:** в токенайзере КАЖДЫЙ фрагмент (и совпавший токен, и «обычный» символ)
  прогонять через `escapeHtml` перед вставкой в HTML.

### Регэкспы токенайзера «съедают» текст не с той позиции
- **Причина:** обычный (нелипкий) регэксп ищет совпадение где угодно дальше по строке.
- **Решение:** использовать **липкий** флаг `/y` и перед каждым `exec` ставить
  `re.lastIndex = i`; тогда совпадение только строго с позиции `i`. Не забыть
  `&& m[0].length > 0`, иначе пустое совпадение зациклит проход.

### Tab уводит фокус вместо вставки табуляции
- **Причина:** по умолчанию Tab переключает фокус между элементами.
- **Решение:** на `keydown` при `e.key === 'Tab'` вызвать `e.preventDefault()` и
  вставить `\t`. Лучше через `document.execCommand('insertText', false, '\t')` —
  он сохраняет нативный Undo и сам шлёт `input`. Фолбаком — ручная правка `value`
  + `dispatchEvent(new Event('input'))`, иначе подсветка/автосохранение не обновятся.

## Emmet (свой мини-движок)

### Tab по аббревиатуре «съел» соседнюю готовую разметку
- **Причина:** при извлечении аббревиатуры слева от каретки `>` — это оператор Emmet,
  и наивный скан утаскивает кусок реального тега (`<br>` → «br>» → `<<br>>`).
- **Решение:** в `extractAbbreviation` останавливаться на пробеле/`\n`, а при встрече
  `<` считать, что задели разметку, и вернуть ПУСТУЮ аббревиатуру (→ обычный таб).
  Скобки `[] {} ()` учитывать счётчиком глубины (внутри них пробелы допустимы).

### Раскрытие «съедает» обычные слова или невалидный ввод
- **Причина:** Tab по любому тексту пытается раскрыть всё подряд.
- **Решение:** `isExpandable()` — раскрывать только если начинается с буквы/`.`/`#`/`(`,
  не содержит `<`, и либо есть структурные символы `>+^*.#[]{}()`, либо это слово —
  ИЗВЕСТНЫЙ HTML-тег (`KNOWN_TAGS`). Иначе обычный таб.

### После раскрытия слетает история отмены / не обновляется подсветка
- **Причина:** прямая запись `textarea.value` рвёт нативный Undo и не шлёт `input`.
- **Решение:** вставлять результат через `document.execCommand('insertText', …)` —
  он сохраняет Undo и сам диспатчит `input` (→ подсветка + автосохранение). Фолбэк —
  ручная правка + `dispatchEvent('input')`. Курсор ставить по метке из `expand()`,
  не забыв прибавить добавленные отступы строк (`newlines * pad.length`).

### Зацикливание токенайзера на пустом совпадении
- **Причина:** элемент-цикл не сдвинул `pos`.
- **Решение:** если внутренний цикл не продвинулся (`pos===start`) — `pos++` принудительно.
  Повторители `*N` ограничивать (`MAX_REPEAT`), иначе `div*999999` повесит вкладку.

### Превью-подсказка стоит не у курсора / «прыгает» при скролле
- **Причина:** в `<textarea>` нет API пиксельных координат курсора; наивно взять
  позицию строки нельзя. А ещё окно превью при прокрутке остаётся на месте.
- **Решение:** координаты курсора считать «зеркалом» — невидимый `div` с теми же
  метриками (`MIRROR_PROPS`: шрифт, padding, border, `tab-size`, `white-space:pre`),
  текст до курсора + span-маркер, читать `offsetLeft/offsetTop`. Экранная позиция =
  `textarea.getBoundingClientRect()` + offset − `scrollTop/Left`. На `scroll`
  пересчитывать позицию (или прятать). Клампить в окно; не влезает вниз → над строкой.

### Превью перехватывает клики/ввод
- **Причина:** окно поверх редактора ловит мышь и мешает печатать/выделять.
- **Решение:** `pointer-events: none` у `#emmet-preview`. Код внутри красить через
  `highlight()` (как в редакторе), вставлять во `<code>` (там уже есть `escapeHtml`).

## Встроенная консоль (мост iframe → платформа)

### `console.log` из кода ребёнка не виден на платформе
- **Причина:** код выполняется в изолированном iframe; его `console` пишет только в
  DevTools родителя нет доступа к нему напрямую.
- **Решение:** в документ iframe внедрить `CONSOLE_HOOK` ДО пользовательского скрипта —
  он подменяет `console.*` и шлёт `parent.postMessage({__console:true,…})`. Платформа
  слушает `message`. Требует `allow-same-origin` + `allow-scripts` в sandbox.

### Чужие/посторонние `message` попадают в консоль
- **Причина:** на `window` могут прилетать сообщения от расширений/других фреймов.
- **Решение:** в обработчике проверять `e.source === preview.contentWindow` и
  `e.data.__console === true`. Текст выводить через `textContent` (не `innerHTML`) —
  иначе вывод `console.log('<img onerror=…>')` исполнится в самой платформе (XSS).

### Консоль бесконечно растёт / показывает старый вывод
- **Причина:** строки копятся и не чистятся между запусками.
- **Решение:** лимит строк (срезать `firstElementChild`), автоскролл вниз,
  очистка при каждом «Запустить» (`updateIframe → clearConsole`).

## Заполнение пропусков (метки [ВПИШИ …])

### Подсветка метки `padding`/`font-weight` «разъезжает» код с textarea
- **Причина:** слой подсветки (`<pre>`) лежит ПОД прозрачным `<textarea>` с
  попиксельно совпадающими метриками. Любое свойство токена-`<span>`, меняющее
  ширину текста (`padding`, `border`, `font-weight: bold`, `letter-spacing`),
  сдвигает символы в слое подсветки относительно textarea — курсор и подсветка
  перестают совпадать.
- **Решение:** для `.tok-todo` (и любых токенов) менять только `color`,
  `background` и `border-radius` — они не влияют на раскладку. Подсветка
  заметна и без жирного шрифта/отступов.

### Метка `[ВПИШИ …]` не подсвечивается / съедается другим токеном
- **Причина:** ни один из базовых паттернов (tag/attr/string/number…) не
  матчит `[`, поэтому без отдельного правила метка красилась бы посимвольно
  (как обычный текст). А если метка стоит ВНУТРИ строки (JS: `console.log("[ВПИШИ…]")`),
  паттерн строки поглощает её целиком — отдельно метка не выделится.
- **Решение:** добавить `TODO_PATTERN` (`/\[ВПИШИ[^\]]*\]/y`) ПЕРВЫМ в каждый
  набор паттернов. В тексте (вне строк) метка подсветится; внутри JS-строки она
  остаётся «строкой» (это допустимо — текст всё равно читается). Поэтому метки
  ставим в текстовых узлах HTML, а не в значениях атрибутов.

## Многошаговый урок (кнопки вставки частей блока)

### `execCommand('insertText')` не вставляет код в скрытый редактор
- **Причина:** execCommand работает с активным (сфокусированным) элементом, а
  `focus()` на textarea внутри скрытой панели (`hidden` → display:none) молча
  игнорируется. Кнопки вставки нажимают на вкладке «📖 Урок», когда все
  редакторы скрыты — вставка «в фоне» невозможна.
- **Решение:** перед вставкой переключить вкладку на целевой файл
  (`switchTab(file)`), затем `focus()` + `setSelectionRange` + execCommand
  (см. `insertAsUserInput`). Бонус: ребёнок видит, куда лёг код; тост
  напоминает вернуться на вкладку урока за следующей частью.

### Ctrl+Z после вставки блока не убирает вставленный код
- **Причина:** прямая запись `editor.value = …` НЕ попадает в нативную историю
  отмены textarea (и вообще стирает её) — вставка кнопкой была «неотменяемой»,
  в отличие от обычного ввода.
- **Решение:** вставлять как пользовательский ввод: выделить заменяемый диапазон
  (`setSelectionRange(start, end)` — например, хвостовые пустые строки) и вызвать
  `execCommand('insertText', false, text)`. Замена выделения + вставка ложатся в
  Undo **одним шагом**, Ctrl+Z возвращает всё как было. Тот же приём, что у
  Tab/Emmet. Фолбэк — ручная правка + `dispatchEvent('input')` (без Undo).
  Примечание: отмена кода НЕ снимает отметку «готово» у части — для возврата
  кода есть кнопка «↻ Вставить … снова».

### Картинки товаров через внешний URL ломают принцип «только локально»
- **Причина:** `https://via.placeholder.com/...` и подобные требуют сети и CDN,
  чего ТЗ запрещает; офлайн карточки будут с битыми картинками.
- **Решение:** использовать **эмодзи** как «картинку» товара (большой `font-size`
  в `.card-img`). Полностью локально и весело менять в задании «сделай сам».

### Вставка блока в `value` рвёт подсветку/автосейв
- **Причина:** прямая запись `editor.value` не диспатчит `input`, поэтому слой
  подсветки и localStorage не обновятся.
- **Решение (актуальное):** вставлять через `execCommand('insertText')`
  (`insertAsUserInput`) — он сам диспатчит `input` → подсветка + автосейв, и
  заодно сохраняет Undo (см. запись про Ctrl+Z выше). Если используется фолбэк
  с ручной правкой `value` — обязательно `dispatchEvent(new Event('input'))`.

### После вставки блок нельзя добавить заново (если ребёнок удалил его код)
- **Причина:** выполненная кнопка делалась `disabled` навсегда. Если ребёнок
  случайно удалил код блока, вернуть его можно было только полным «Начать заново».
- **Решение:** выполненный блок оставляем кликабельным — кнопка переходит в
  «↻ Вставить блок N снова» (класс `.done`, НЕ disabled) и повторно дописывает
  сниппет. Линейный порядок (`firstUndoneStep`) применяется только к ещё не
  добавленным блокам; повторная вставка разрешена в любой момент.

### HTML-блок дописывается после `</body></html>` → сломанная структура
- **Причина:** простой append кладёт код в самый конец файла. Если ученик
  предварительно создал каркас (Emmet «!»), блок окажется ПОСЛЕ `</body></html>`.
- **Решение:** HTML вставлять перед `</body>` (`insertHtmlBeforeBody`, regex
  `/([ \t]*)<\/body>/i`, отступ на уровень глубже). Если `</body>` нет — НЕ
  вставлять, показать тост «Сначала создай каркас сайта» (через Emmet «!»). CSS/JS
  это не касается — они отдельные файлы и дописываются в конец.

### Полный документ ученика, обёрнутый ещё раз, даёт вложенные `<html>/<body>`
- **Причина:** `updateIframe` всегда оборачивал содержимое HTML-редактора в свой
  `<body>`. Когда там уже полный каркас (Emmet «!»), получается документ внутри
  документа.
- **Решение:** в `updateIframe` две ветки. Есть `</body>` → берём как полный
  документ, внедряя `<style>` перед `</head>` и скрипты перед `</body>`. Нет →
  оборачиваем как фрагмент (старое поведение).

### `String.replace` портит код с символом `$` (`$&`, `$1`, `$\``)
- **Причина:** второй аргумент-строка у `replace` трактует `$`-подстановки. Если
  в CSS/JS ребёнка есть `$` (напр. `"$&"`), при вставке в документ он исказится.
- **Решение:** использовать **функцию-replacer** (`() => replacement`) — она
  возвращает строку дословно, без `$`-магии.

### Вставка блоков не по порядку даёт неверный визуальный порядок
- **Причина:** append дописывает в конец; если нажать «блок 3» раньше «блока 1»,
  товары окажутся выше шапки.
- **Решение:** линейная разблокировка — активна только кнопка `firstUndoneStep()`;
  остальные disabled. `insertBlock` дополнительно игнорирует вызов не по порядку.

### Инлайн-`код` в markdown-полях ломает JS-строку
- **Причина:** если поле шага — шаблонный литерал (бэктики), то ` `code` ` внутри
  закроет литерал.
- **Решение:** короткие markdown-поля писать в **двойных кавычках** (бэктики там
  литеральны, переносы через `\n`); многострочные сниппеты — в шаблонных литералах
  (в них следить за отсутствием `` ` `` и `${`). Кавычки внутри инлайн-кода
  экранировать `\"`.

## Открытие сайта в отдельной вкладке

### Вкладка с Blob-URL не отражает правки и «ломается» после revoke
- **Причина:** `window.open(URL.createObjectURL(blob))` открывает СТАТИЧНЫЙ снимок
  кода. Он не обновляется при правках, а `revokeObjectURL` (через таймаут) делает
  URL недействительным — перезагрузка/повторное открытие вкладки дают пустую
  страницу («перестаёт работать»).
- **Решение:** открывать отдельный файл `preview.html` — живое зеркало. Платформа
  на каждый «Запустить» пишет готовый документ в localStorage (`previewDoc`);
  `preview.html` читает его в `iframe.srcdoc` и подписан на событие `storage` →
  обновляется синхронно с «Запустить», переживает перезагрузку, ничего не
  отзывается. Повторный клик переиспользует именованную вкладку (`'shopPreview'`).

## Редактор: отмена (Ctrl+Z) и перезагрузка

### Ctrl+Z не откатывает правки, сделанные до перезагрузки страницы
- **Причина:** нативная история отмены `<textarea>` живёт только в пределах
  одной загрузки и **стирается при перезагрузке**. Контент мы восстанавливаем
  присваиванием `textarea.value = …`, а такое присваивание ещё и обнуляет стек
  отмены. Засеять нативную историю программно нельзя.
- **Решение:** ведём СВОЙ стек снимков на каждый редактор и храним его в
  localStorage (`undoHistory`). `histRecord` пишет шаги на `input` (с коалесингом
  набора), Ctrl+Z/Ctrl+Shift+Z/Ctrl+Y перехватываем в `keydown` до нативной
  отмены (`histUndo`/`histRedo`), на старте поднимаем стек (`histLoad`). Так
  отмена работает и после перезагрузки. Запись дебаунсится + флаш на
  `beforeunload`; при переполнении квоты стек ужимается. `resetToTemplate`
  заводит историю заново.

### Отступ у перенесённых строк (wrappingIndent) — НЕ делаем
- **Причина:** видимый код рисует слой `.highlight` (`<pre>`), а курсор/выделение
  — прозрачный `<textarea>` поверх. Перенос делает нативный wrap textarea, и он
  НЕ умеет индентировать перенесённые (visual-continuation) строки; CSS для этого
  нет (`text-indent … each-line`/`hanging` браузеры не поддерживают). Если
  индентировать только подсветку — она начнёт переноситься в других местах, чем
  textarea, и курсор/подсветка/нумерация разъедутся по обеим осям.
- **Решение (по согласованию с пользователем, 2026-06):** перенесённый «хвост»
  строки оставляем прижатым влево (без отступа по indent). Так курсор, подсветка
  и номера всегда совпадают. Не пытаться добавить отступ переноса без ухода от
  textarea-оверлея (например, на contenteditable) — это отдельная большая задача.

## Раскладка из трёх колонок (перетаскивание ширины)

### Перетаскивание разделителя «срывается» при заходе курсора на iframe превью
- **Причина:** `iframe` поглощает события мыши — `mousemove`/`pointermove`,
  слушаемые на разделителе или `document`, перестают приходить, как только
  курсор оказывается над превью. Ползунок «отлипает».
- **Решение:** использовать Pointer Events с **захватом указателя**
  (`splitter.setPointerCapture(e.pointerId)` на `pointerdown`) — события
  `pointermove` продолжают идти на разделитель поверх iframe. Дополнительно на
  время перетаскивания вешать `body.resizing` с `#preview{pointer-events:none}`
  и `user-select:none`.

### Не скроллится задание: CSS-селектор `#task.md` молча не матчит
- **Причина:** у элемента `id="task.md"` точка в id. В CSS `#task.md` парсится
  как «id=`task` И класс `md`» — такого элемента нет, правило не применяется.
  После выноса задания в свою колонку `overflow:auto` навесили через `#task.md`,
  и вертикальный скролл пропал (раньше прокрутка шла через класс `.tab-pane`).
- **Решение:** не таргетить этот элемент через `#id`. Либо экранировать точку
  (`#task\.md`), либо целиться по классу — `.col-task .markdown-body`
  (как сделано). То же касается любых id с `.`/`:` в значении.

### Колонки на фиксированной ширине (px) не подстраиваются под размер окна
- **Причина:** жёсткие `width: Npx` у колонок не масштабируются при изменении
  размера окна и ломают сумму ширин.
- **Решение:** задавать ширину через `flex-grow` при `flex-basis:0` (тогда
  `grow` ≡ доля ширины). Перетаскивание перераспределяет `grow` только между
  двумя соседними колонками, сохраняя их сумму — остальные не трогаются.
  Адаптив (`<800px`) перекрывает inline `flex-grow` через `flex:none !important`.

## localStorage

### Исключение при доступе к localStorage (приватный режим / отключённый storage)
- **Причина:** в некоторых режимах обращение к `localStorage` кидает `SecurityError`.
- **Решение:** оборачивать чтение/запись в `try/catch`; при ошибке вести себя как «данных нет».

### После «Сброса» старый код возвращается после перезагрузки
- **Причина:** сбросили textarea, но не удалили ключ из localStorage.
- **Решение:** в `resetToTemplate()` обязательно `localStorage.removeItem('savedLessonCode')`
  (или перезапись initial-значениями) перед/после обновления редакторов.

## Скачивание проекта (ZIP вручную, без библиотек)

### Архив не открывается / «повреждён»
- **Причина:** неверный CRC32, рассинхрон размеров, неправильное смещение локального заголовка в центральном каталоге, или забытый EOCD.
- **Решение:** строго соблюдать формат. Сигнатуры: локальный `0x04034b50`, центральный `0x02014b50`, EOCD `0x06054b50`. Метод `0` (store), сжатый размер = исходному. `localOffset` каждого файла = его смещение **в потоке локальных заголовков**, считается ДО добавления центрального каталога. EOCD: `centralOffset` = суммарная длина всех локальных частей. Проверка: `zipfile.ZipFile(...).testzip()` должен вернуть `None`.

### Кракозябры в именах/содержимом с кириллицей
- **Причина:** ZIP по умолчанию трактует имена как CP437.
- **Решение:** ставить флаг бит 11 (`0x0800`) в general purpose flags локального и центрального заголовков — объявляет имена в UTF-8. Имена и данные кодировать через `TextEncoder` (UTF-8). Размеры брать в **байтах**, а не в `length` строки (символ ≠ байт).

### CRC32 без таблицы даёт неверный результат
- **Причина:** забыт начальный/финальный XOR с `~0`, или потеря старших битов (signed-сдвиги).
- **Решение:** `crc=~0; crc=(crc>>>1)^(0xEDB88320 & -(crc&1)); ... return (~crc)>>>0`. Беззнаковые сдвиги (`>>>`) обязательны.

## Скачивание файлов

### Память течёт / ссылка не скачивается
- **Причина:** не освобождён objectURL или ссылка не добавлена в DOM в некоторых браузерах.
- **Решение:** `URL.createObjectURL(blob)` → создать `<a>` с `download`, добавить в DOM,
  `click()`, удалить из DOM, затем `URL.revokeObjectURL(url)`.

## Прочее

### `defer` vs порядок инициализации
- **Причина:** если `main.js` без `defer` и подключён в `<head>`, DOM ещё не готов.
- **Решение:** подключать с `defer` или перед `</body>`; всю инициализацию — в обработчике `DOMContentLoaded`.
